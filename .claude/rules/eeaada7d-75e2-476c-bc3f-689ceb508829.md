<rule_activation id="eeaada7d-75e2-476c-bc3f-689ceb508829" title="Adopt Server-Only Data Access Layer with Transactional Boundaries: Complex Operations Involving" applies_to="**/*">
These rules are ALWAYS ACTIVE for all server-side data access operations. All data mutations and queries that interact with the database MUST follow the patterns defined herein.
</rule_activation>

### Rules

- **R-DAL-001** SHOULD: Complex operations involving multiple entities SHOULD use database transactions to ensure atomicity and data consistency.
- **R-DAL-002** MUST: All database queries and mutations MUST be executed server-side through the server-only layer, never directly from client-side code.
- **R-DAL-003** MUST: All data mutation operations (create, update, delete) MUST be implemented as exported functions in the server-only layer with proper error handling.
- **R-DAL-004** MUST: Server-only data access functions MUST define clear input/output types using TypeScript for compile-time validation.
- **R-DAL-005** MUST: All server-only functions MUST implement a consistent error handling strategy using typed exceptions or result types.
- **R-DAL-006** SHOULD: Server-only data access functions SHOULD be organized by domain entity in a clear directory structure (e.g., server-only/folder/, server-only/recipient/, server-only/profile/).
- **R-DAL-007** SHOULD: Shared utilities for common patterns like transaction management, authorization checks, and input validation SHOULD be created to reduce boilerplate.
- **R-DAL-008** MUST: Authentication and authorization checks MUST be enforced for all data access operations.
- **R-DAL-009** MUST: Data validation and transformation logic MUST occur before persistence.

### Verify

```bash
# Check for client-side database imports outside server-only layer
grep -r 'import.*prisma\|import.*db' apps/*/app --exclude-dir=node_modules | grep -v 'server-only' | grep -v '.server.' || echo 'No client-side database imports detected'

# Verify minimum number of server-only modules exist
find packages/lib/server-only -type f -name '*.ts' | wc -l | awk '{if($1 >= 5) print "PASS: Found " $1 " server-only modules"; else print "FAIL: Expected at least 5 server-only modules"}'

# Verify data mutation functions are exported from server-only layer
grep -r 'export.*function.*delete\|export.*function.*create\|export.*function.*update' packages/lib/server-only --include='*.ts' | wc -l | awk '{if($1 >= 3) print "PASS: Found " $1 " data mutation functions"; else print "FAIL: Expected at least 3 mutation functions"}'
```

**Accept when:**
- No client-side code directly imports database clients or executes queries; all database access is mediated through server-only layer
- At least 5 server-only data access modules exist, organized by domain entity, each handling specific business operations
- All data mutation operations (create, update, delete) are implemented as exported functions in the server-only layer with proper error handling
- TypeScript types are defined for all server-only function inputs and outputs
- Consistent error handling strategy is implemented across all server-only modules
- Authentication and authorization checks are present for all data access operations

Claude Code MUST NOT skip or defer verification. All data access patterns MUST be verified against the rules before code generation or modification. Violations detected by linting rules or CI/CD checks MUST result in build failures and require remediation.
