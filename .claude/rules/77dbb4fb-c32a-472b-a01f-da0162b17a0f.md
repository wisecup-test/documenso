<rule_activation id="77dbb4fb-c32a-472b-a01f-da0162b17a0f" title="Adopt Server-Only Module Pattern for External API Integrations: Server Only Directory" applies_to="**/*">
These rules are ALWAYS ACTIVE for all files in the codebase to enforce strict isolation of server-only modules from client-side code.
</rule_activation>

### Rules

- **R-SERVERONLY-001** SHOULD: Server-only directory structures SHOULD follow the pattern `packages/lib/server-only/{domain}/` for clear organizational boundaries.

### Verify

```bash
# Check for client-side imports of server-only modules in Remix routes
grep -r "from.*server-only" apps/remix/app/routes --include="*.tsx" --include="*.ts" | grep -v "loader\|action" || echo "No client-side server-only imports detected"

# Count server-only modules that export sensitive credentials
find packages/lib/server-only -type f -name "*.ts" -exec grep -l "export.*API_KEY\|SECRET\|PASSWORD" {} \; | wc -l

# Verify client bundle size and check for server-only code leakage
npm run build && du -sh dist/client/**/*.js | awk '{if($1 ~ /M/ && $1+0 > 5) print "Warning: Large client bundle detected: " $0}'
```

**Accept when:**
- All server-only modules reside under designated `packages/lib/server-only/` directory structures and are not imported directly in client-side code
- Build process successfully prevents bundling of server-only code in client JavaScript outputs
- CI pipeline includes automated checks that fail on detection of server-only imports in client contexts
- All external API integrations, PDF operations, and audit log rendering are implemented as server-only modules
- No sensitive credentials (API_KEY, SECRET, PASSWORD) are exported from server-only modules

<enforcement>
Claude Code MUST NOT skip or defer verification. All server-only module patterns MUST be validated through automated CI checks, ESLint rules, and build-time bundle analysis before code reaches production.
</enforcement>
