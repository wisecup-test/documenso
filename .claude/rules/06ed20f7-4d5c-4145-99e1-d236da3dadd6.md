<rule_activation id="06ed20f7-4d5c-4145-99e1-d236da3dadd6" title="Standardize Cache Layer Configuration for Runtime Data Management: Cache Keys Deterministic" applies_to="**/*">
These rules are ALWAYS ACTIVE for all runtime components that manage data caching, including API handlers, route handlers, and data fetching utilities. All new cache implementations MUST follow these patterns.
</rule_activation>

### Rules

- **R-37-001** MUST: Cache keys MUST be deterministic and include all parameters that affect the cached result to prevent cache collision.
- **R-37-004** MUST: Enforce TTL configuration for all cached data to prevent memory exhaustion from unbounded cache growth.
- **R-37-006** MUST: Enforce co-location of cache invalidation with mutations to prevent stale data from being served after mutations.

### Verify

```bash
# Check for cache configuration patterns in route handlers and API utilities
grep -r 'cache.*config\|cacheConfig\|Cache.*Options' apps/remix/app/routes packages/trpc/utils --include='*.ts' --include='*.tsx' | wc -l

# Check for TTL configuration patterns
grep -r 'TTL\|timeToLive\|maxAge' apps/remix/app/routes packages/trpc/utils --include='*.ts' --include='*.tsx' | grep -v node_modules | wc -l

# Check for cache invalidation patterns
grep -r 'invalidate.*cache\|cache.*invalidate\|revalidate' apps/remix/app/routes packages/trpc/utils --include='*.ts' --include='*.tsx' | wc -l
```

**Accept when:**
- All route handlers and API utilities that fetch data have explicit cache configuration declarations (grep count > 0 for cache config patterns)
- Cache invalidation logic is present alongside mutation operations (grep count shows invalidation patterns in mutation files)
- TTL configuration is explicitly set for cached data rather than using framework defaults (grep count shows TTL patterns in cache configurations)
- Code review checklist includes verification of cache key uniqueness and invalidation strategy
- Cache keys include all parameters that affect the cached result (user ID, team ID, resource ID, query parameters)
- No cache stampede vulnerabilities exist; request coalescing or cache locking is implemented for high-traffic scenarios
- Cache versioning or tags are implemented for complex invalidation scenarios
- LRU eviction policies are configured for long-running processes

<enforcement>
Claude Code MUST NOT skip or defer verification of cache configuration patterns. All data-fetching code MUST include explicit cache configuration with deterministic keys and TTL values. Cache invalidation logic MUST be co-located with mutations. Violations detected during verification MUST be flagged for code review rejection or technical debt tracking.
</enforcement>
