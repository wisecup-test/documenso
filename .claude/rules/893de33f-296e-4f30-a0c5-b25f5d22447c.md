<rule_activation id="893de33f-296e-4f30-a0c5-b25f5d22447c" title="Enforce Server-Only Module Isolation for Secrets Management: Environment Variable Access" applies_to="**/*">
These rules are ALWAYS ACTIVE for all server-side code that handles sensitive configuration, environment variables, or secrets management operations.
</rule_activation>

### Rules

- **R-SOI-001** SHOULD: Environment variable access patterns SHOULD be centralized in dedicated configuration modules within server-only boundaries.

### Verify

```bash
# Verify server-only imports are present in secrets management modules
grep -r "import.*server-only" packages/lib/server-only/ | wc -l

# Detect process.env access outside server-only modules (excluding NEXT_PUBLIC_ prefixed variables)
grep -r "process\.env" packages/ --include="*.ts" --include="*.tsx" | grep -v "server-only" | grep -v "NEXT_PUBLIC" | wc -l

# Run build and bundle analysis
npm run build && npx @next/bundle-analyzer --analyze
```

**Accept when:**
- All files in server-only directories contain 'import "server-only"' statement
- No process.env access for secrets exists outside server-only modules (excluding NEXT_PUBLIC_ prefixed variables)
- Client bundle analysis shows no environment variable access or credential-related code
- Build fails with clear error message when client code attempts to import server-only modules

<enforcement>
Claude Code MUST NOT skip or defer verification. Automated linting rules, CI pipeline bundle analysis, code review checklists, and periodic security audits are mandatory to enforce server-only module isolation boundaries.
</enforcement>
