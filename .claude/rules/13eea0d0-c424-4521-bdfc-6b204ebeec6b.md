<rule_activation id="13eea0d0-c424-4521-bdfc-6b204ebeec6b" title="Implement Cache Layer for Public API Route Handlers: Authenticated Route Handlers" applies_to="apps/remix/app/routes/_authenticated+/**/*.{ts,tsx}">
These rules are ALWAYS ACTIVE for all authenticated route handlers serving organization, team, and admin-level data through public-facing APIs in the Remix application.
</rule_activation>

### Rules

- **R-CACHE-001** MUST: All authenticated route handlers serving organization, team, and admin data MUST implement a cache layer for data retrieval operations.
- **R-CACHE-002** MUST: Cache keys MUST include scope identifiers (orgUrl, teamUrl, userId) to ensure tenant isolation in multi-tenant scenarios.
- **R-CACHE-003** MUST: Cache invalidation logic MUST be present in mutation operations (create, update, delete) that automatically clear related cache entries.
- **R-CACHE-004** SHOULD: Use a shared cache utility module (e.g., `app/utils/cache.server.ts`) that provides consistent cache access patterns and key generation functions.
- **R-CACHE-005** SHOULD: Implement cache key naming convention: `{scope}:{resource}:{identifier}` (e.g., `org:members:acme-corp`, `team:settings:engineering-team`).
- **R-CACHE-006** SHOULD: Add cache hit/miss metrics to application monitoring to track cache effectiveness and identify routes needing TTL adjustments.
- **R-CACHE-007** SHOULD: Document cache behavior in route handler comments, including TTL values and invalidation triggers.
- **R-CACHE-008** MAY: Implement circuit breaker pattern around cache access to gracefully degrade to direct database queries on cache infrastructure failure.

### Exceptions

- **EXC-001**: Route handles real-time collaborative features where stale data would cause user confusion or data conflicts.
- **EXC-002**: Regulatory or compliance requirements mandate fresh data retrieval for audit trails.

### Verify

```bash
# Count cache implementation usage in authenticated routes
grep -r "cache" apps/remix/app/routes/_authenticated+ --include="*.tsx" --include="*.ts" | grep -E "(useCache|cacheGet|cacheSet)" | wc -l

# Count cache invalidation logic in authenticated routes
grep -r "invalidate" apps/remix/app/routes/_authenticated+ --include="*.tsx" --include="*.ts" | grep -E "(cache|Cache)" | wc -l

# Count shared cache utility modules
find apps/remix/app/utils -name "*cache*.ts" -o -name "*cache*.server.ts" | wc -l
```

**Accept when:**
- At least 80% of authenticated route handlers in the identified files implement cache layer for data retrieval operations
- A shared cache utility module exists and is imported by multiple route handlers
- Cache invalidation logic is present in mutation operations for cached resources
- Cache keys include scope identifiers (orgUrl, teamUrl, userId) to ensure tenant isolation
- Cache behavior is documented in route handler comments with TTL values and invalidation triggers

<enforcement>
Claude Code MUST verify all rules before approving changes to authenticated route handlers. Cache implementation and invalidation logic MUST be present in all new or modified routes serving organization, team, or admin data. Violations require justification or implementation of cache layer. Performance regressions in monitored routes trigger investigation of cache effectiveness.
</enforcement>
