<rule_activation id="4163b7b8-cae1-4958-a5cd-2270c2d5edc8" title="Establish Type-Safe Public API Contracts for Library Modules: Contract Definitions Located" applies_to="**/*">
These rules are ALWAYS ACTIVE for all public API contract definitions and library module interfaces within the codebase, including all packages under /packages/, public API endpoints in /packages/api/, shared library modules in /packages/lib/, test constants and fixtures in /packages/app-tests/, and OpenAPI specifications.
</rule_activation>

### Rules

- **R-CONTRACT-001** SHOULD: Contract definitions SHOULD be co-located with their implementation within the same package structure.

### Verify

```bash
# Count contract definitions across packages
grep -r "export.*contract" packages/*/contract.ts | wc -l

# Find OpenAPI and contract files with exports
find packages/api -name 'openapi.ts' -o -name 'contract.ts' | xargs grep -l 'export'

# Verify TypeScript strict mode compilation
tsc --noEmit --strict && echo 'Type checking passed'

# Identify server-only modules
grep -r "server-only" packages/lib | grep -v node_modules
```

**Accept when:**
- All public API endpoints have corresponding TypeScript contract definitions with complete type coverage
- TypeScript strict mode compilation passes without type errors in contract files
- OpenAPI specifications exist for versioned API endpoints (v1, v2, etc.)
- Server-only modules are properly isolated and cannot be imported by client-side code

<enforcement>
Claude Code MUST verify all rules during code review and MUST NOT skip or defer verification. TypeScript compiler in strict mode, ESLint rules, code review checklists, and automated tests are mandatory enforcement mechanisms. CI builds MUST fail if violations occur. Pull requests MUST be blocked if new public APIs lack contract definitions.
</enforcement>
