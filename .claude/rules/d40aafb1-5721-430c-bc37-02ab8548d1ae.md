<rule_activation id="d40aafb1-5721-430c-bc37-02ab8548d1ae" title="Establish Type-Safe Public API Contracts for Library Modules: Public Provide Openapi" applies_to="**/*">
These rules are ALWAYS ACTIVE for all public API contract definitions and library module interfaces within the codebase, particularly in /packages/ directories, public API endpoints in /packages/api/, shared library modules in /packages/lib/, and test constants in /packages/app-tests/.
</rule_activation>

### Rules

- **R-API-001** SHOULD: Public APIs SHOULD provide OpenAPI/Swagger specifications for documentation and validation.

### Verify

```bash
# Count contract definitions across packages
grep -r "export.*contract" packages/*/contract.ts | wc -l

# Find OpenAPI and contract files with exports
find packages/api -name 'openapi.ts' -o -name 'contract.ts' | xargs grep -l 'export'

# Verify TypeScript strict mode compilation passes
tsc --noEmit --strict && echo 'Type checking passed'

# Identify server-only modules
grep -r "server-only" packages/lib | grep -v node_modules
```

**Accept when:**
- All public API endpoints have corresponding TypeScript contract definitions with complete type coverage
- TypeScript strict mode compilation passes without type errors in contract files
- OpenAPI specifications exist for versioned API endpoints (v1, v2, etc.)
- Server-only modules are properly isolated and cannot be imported by client-side code

<enforcement>
Claude Code MUST NOT skip or defer verification. TypeScript compiler in strict mode, ESLint rules, code review checklists, and automated tests are mandatory enforcement mechanisms. CI builds MUST fail if TypeScript compilation errors occur in contract files or if new public APIs lack contract definitions.
</enforcement>
