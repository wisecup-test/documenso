<rule_activation id="9da37add-1d78-47a2-baad-c62c7d3f9d86" title="Adopt E2E API Testing for Frontend Rendering Model Validation: Endpoints That Provide" applies_to="**/*">
These rules are ALWAYS ACTIVE for all API endpoints and test files. E2E API testing is mandatory for all endpoints that provide data for frontend rendering across REST (v1, v2) and tRPC APIs.
</rule_activation>

### Rules

- **R-E2E-001** MUST: All API endpoints that provide data for frontend rendering MUST have corresponding E2E test specifications.
- **R-E2E-002** MUST: E2E tests MUST cover all REST API endpoints (v1, v2) that serve frontend applications.
- **R-E2E-003** MUST: E2E tests MUST cover all tRPC procedures that provide data for UI rendering.
- **R-E2E-004** MUST: E2E tests MUST cover API endpoints for document search, retrieval, and management.
- **R-E2E-005** MUST: E2E tests MUST cover API endpoints for template operations and field prefilling.
- **R-E2E-006** MUST: E2E tests MUST cover API endpoints for envelope distribution and validation.
- **R-E2E-007** MUST: E2E tests MUST cover API endpoints for embedding and presigning operations.
- **R-E2E-008** SHOULD: Organize E2E tests in directory structure matching API versioning: e2e/api/v1/, e2e/api/v2/, e2e/api/trpc/.
- **R-E2E-009** SHOULD: Use descriptive test names that clearly indicate the API endpoint and scenario being tested.
- **R-E2E-010** SHOULD: Implement test data factories to generate consistent, realistic test data for API requests.
- **R-E2E-011** SHOULD: Create shared test utilities for common operations like authentication, test data setup, and response validation.
- **R-E2E-012** MAY: Use API client libraries in tests to ensure tests exercise the same code paths as production frontend applications.

### Exceptions

- **EXC-001**: API endpoints are deprecated and scheduled for removal within 30 days.
- **EXC-002**: Internal admin APIs with no direct frontend consumption.

### Verify

```bash
# Count E2E API test specification files
find packages/app-tests/e2e/api -name '*.spec.ts' | wc -l | grep -E '^[8-9]|^[1-9][0-9]+$'

# Count test cases across all E2E API tests
grep -r 'describe\|test\|it' packages/app-tests/e2e/api --include='*.spec.ts' | wc -l

# List all E2E API test files
npm test -- --testPathPattern=e2e/api --listTests | wc -l
```

**Accept when:**
- At least 8 E2E API test specification files exist covering major API endpoints
- All API versions (v1, v2, tRPC) have corresponding test coverage for frontend-facing endpoints
- E2E test suite runs successfully in CI/CD pipeline with >90% pass rate
- Test coverage includes document operations, template management, envelope workflows, and search functionality
- E2E tests validate the complete request-response cycle including authentication, data transformation, and response formatting
- Test file naming conventions follow the pattern: `{endpoint-name}.spec.ts`
- Test organization matches API versioning structure

<enforcement>
Claude Code MUST NOT skip or defer verification of E2E API test coverage. All API endpoints providing data for frontend rendering MUST have corresponding E2E test specifications before code is approved. Violations block pull request merging and trigger CI/CD pipeline failures.
</enforcement>
