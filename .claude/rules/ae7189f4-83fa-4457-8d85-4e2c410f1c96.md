<rule_activation id="ae7189f4-83fa-4457-8d85-4e2c410f1c96" title="Adopt Server-Only Data Access Layer with Specialized Query Functions: Complex Queries Separated" applies_to="**/*">
These rules are ALWAYS ACTIVE for all server-side data access operations. All database queries MUST be encapsulated in dedicated server-only modules following the established pattern.
</rule_activation>

### Rules

- **R-DAL-001** SHOULD: Complex queries SHOULD be separated into internal and external variants (e.g., find-folders.ts and find-folders-internal.ts) where internal variants bypass certain validations for trusted system operations.
- **R-DAL-002** MUST: All database read operations (SELECT queries) be encapsulated in server-only modules.
- **R-DAL-003** MUST: All database write operations (INSERT, UPDATE, DELETE) be encapsulated in server-only modules.
- **R-DAL-004** MUST: All data access requiring authentication or authorization be encapsulated in server-only modules.
- **R-DAL-005** MUST: Use TypeScript's module resolution to enforce server-only imports and configure bundler to error on server-only imports in client code.
- **R-DAL-006** SHOULD: Establish naming conventions: get-* for single record retrieval, find-* for queries returning multiple records, create-*, update-*, delete-* for mutations, *-internal for trusted system operations.
- **R-DAL-007** SHOULD: Each function export a single default or named function with clear TypeScript types for parameters and return values.
- **R-DAL-008** SHOULD: Include JSDoc comments documenting authorization requirements, expected errors, and usage examples.
- **R-DAL-009** SHOULD: Use a shared database client/connection pool imported by all data access functions to ensure consistent configuration.

### Verify

```bash
# Count server-only module usage
grep -r 'server-only' --include='*.ts' --include='*.tsx' | grep -v 'node_modules' | wc -l

# Find server-only files missing function exports
find . -path '*/server-only/*' -name '*.ts' | xargs grep -L 'export.*function' | wc -l

# Detect direct database imports in client-side code
grep -r 'import.*prisma\|import.*db' --include='*.tsx' --exclude-dir='server-only' --exclude-dir='api' | grep -v 'node_modules' | wc -l
```

**Accept when:**
- All database access operations are located in server-only directories with no direct database imports in client-side code
- Each data access function follows the verb-entity-qualifier naming pattern and exports a single focused operation
- Code review confirms no database credentials or connection strings are exposed outside the server-only layer
- Automated tests verify that client bundles do not include server-only modules
- Build fails for any client-side imports of server-only modules
- Code review rejects direct database access outside server-only layer

Claude Code MUST NOT skip or defer verification. All database access patterns MUST comply with server-only module encapsulation. Violations require security review and refactoring tickets.
