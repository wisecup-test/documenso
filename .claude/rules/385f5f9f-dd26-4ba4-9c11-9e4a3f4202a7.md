<rule_activation id="385f5f9f-dd26-4ba4-9c11-9e4a3f4202a7" title="Enforce Server-Side Environment Variable Validation with Zod Schemas: Configuration Schemas Located" applies_to="**/*.ts,**/*.tsx">
These rules are ALWAYS ACTIVE for all server-side TypeScript/JavaScript code executing in Node.js runtime, including API routes, server-only utilities, background jobs, authentication modules, and database configuration.
</rule_activation>

### Rules

- **R-ENV-001** SHOULD: Configuration schemas SHOULD be co-located with the modules that consume them or centralized in a configuration module.
- **R-ENV-002** MUST: All server-side code MUST use Zod schemas to validate environment variables and configuration inputs at module initialization.
- **R-ENV-003** MUST: Server-only modules MUST import the 'server-only' package at the top of files to enforce that configuration validation code never runs on the client.
- **R-ENV-004** MUST: Direct `process.env` access MUST NOT occur outside of designated configuration modules; all configuration MUST be accessed through validated configuration objects.
- **R-ENV-005** SHOULD: Configuration validation SHOULD use Zod's `.transform()` and `.refine()` methods for complex validation logic like URL parsing, enum validation, or cross-field dependencies.
- **R-ENV-006** SHOULD: Zod schemas SHOULD redact sensitive values in error messages to prevent API keys and secrets from being logged.
- **R-ENV-007** MUST: Application startup MUST fail fast with clear error messages when required environment variables are missing or invalid.

### Verify

```bash
# Check for direct process.env access in server-side code (excluding validated configuration modules)
grep -r "process\.env" --include="*.ts" --include="*.tsx" --exclude-dir=node_modules | grep -v "z\.object\|zod\|schema" | wc -l

# Find server-only modules missing Zod imports
find . -name "*.ts" -path "*/server-only/*" -exec grep -L "import.*zod\|from.*zod" {} \;

# Run configuration and environment tests with coverage
npm test -- --testPathPattern="config|env" --coverage
```

**Accept when:**
- Grep for direct `process.env` access returns 0 instances in server-side code (excluding validated configuration modules)
- All server-only modules import and use validated configuration objects rather than accessing environment variables directly
- Configuration validation tests achieve 100% coverage of all environment variables used in production code
- Application startup fails fast with clear error messages when required environment variables are missing or invalid
- ESLint detects no violations of direct `process.env` access outside of designated configuration modules
- All new environment variable usage includes corresponding Zod validation schemas

Claude Code MUST verify all rules are satisfied before approving server-side code changes. Verification is mandatory and cannot be deferred. ESLint rules MUST be configured to flag violations, and CI/CD pipeline integration tests MUST validate configuration before deployment.
</enforcement>
