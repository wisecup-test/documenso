<rule_activation id="6e6b5637-31b8-4759-8498-6bf8c25eef7c" title="Establish Type-Safe Public API Contracts for Library Modules: Library Modules Exposing" applies_to="**/*">
These rules are ALWAYS ACTIVE for all public API contract definitions and library module interfaces within the codebase.
</rule_activation>

### Rules

- **R-TSAPI-001** MUST: Library modules exposing public functionality MUST declare their public interface in a dedicated contract file (e.g., contract.ts)

### Verify

```bash
# Count contract files in packages
grep -r "export.*contract" packages/*/contract.ts | wc -l

# Find OpenAPI and contract definitions
find packages/api -name 'openapi.ts' -o -name 'contract.ts' | xargs grep -l 'export'

# Verify TypeScript strict mode compilation
tsc --noEmit --strict && echo 'Type checking passed'

# Check for server-only module isolation
grep -r "server-only" packages/lib | grep -v node_modules
```

**Accept when:**
- All public API endpoints have corresponding TypeScript contract definitions with complete type coverage
- TypeScript strict mode compilation passes without type errors in contract files
- OpenAPI specifications exist for versioned API endpoints (v1, v2, etc.)
- Server-only modules are properly isolated and cannot be imported by client-side code

<enforcement>
Claude Code MUST NOT skip or defer verification. TypeScript compiler in strict mode, ESLint rules, code review checklists, and automated tests are mandatory enforcement gates. CI builds MUST fail if violations occur. Pull requests MUST be blocked if new public APIs lack contract definitions.
</enforcement>
