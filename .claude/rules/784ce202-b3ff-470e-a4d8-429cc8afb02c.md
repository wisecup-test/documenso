<rule_activation id="784ce202-b3ff-470e-a4d8-429cc8afb02c" title="Implement Cache Layer for Public API Route Handlers: Sensitive Authentication Tokens" applies_to="**/*">
These rules are ALWAYS ACTIVE for all authenticated Remix route handlers under _authenticated+ directory, organization-level routes (o.$orgUrl.*), team-level routes (t.$teamUrl+/*), admin routes (admin+/*), and settings, members, teams, groups, templates, documents, and support routes.
</rule_activation>

### Rules

- **R-CACHE-001** MUST NOT: Sensitive authentication tokens or credentials MUST NOT be stored in the cache layer.
- **R-CACHE-002** MUST: All authenticated Remix route handlers implement cache layer for data retrieval operations on high-traffic, read-heavy endpoints (members, teams, groups, settings).
- **R-CACHE-003** MUST: Cache keys include scope identifiers (orgUrl, teamUrl, userId) to ensure tenant isolation in multi-tenant scenarios.
- **R-CACHE-004** MUST: Cache invalidation logic is present in mutation operations (create, update, delete) that automatically clear related cache entries.
- **R-CACHE-005** SHOULD: Use a shared cache utility module (e.g., `app/utils/cache.server.ts`) that provides consistent cache access patterns and key generation functions.
- **R-CACHE-006** SHOULD: Implement cache key naming convention: `{scope}:{resource}:{identifier}` (e.g., `org:members:acme-corp`, `team:settings:engineering-team`).
- **R-CACHE-007** SHOULD: Implement circuit breaker pattern around cache access to ensure graceful degradation to direct database queries on cache infrastructure failure.
- **R-CACHE-008** SHOULD: Configure appropriate TTL values for all cached data and implement LRU eviction policies with maximum cache size limits.
- **R-CACHE-009** SHOULD: Document cache behavior in route handler comments, including TTL values and invalidation triggers.
- **R-CACHE-010** MAY: Exception requests for routes handling real-time collaborative features or regulatory/compliance requirements may bypass cache layer with documented justification.

### Verify

```bash
# Count cache implementation patterns in authenticated routes
grep -r "cache" apps/remix/app/routes/_authenticated+ --include="*.tsx" --include="*.ts" | grep -E "(useCache|cacheGet|cacheSet)" | wc -l

# Count cache invalidation patterns
grep -r "invalidate" apps/remix/app/routes/_authenticated+ --include="*.tsx" --include="*.ts" | grep -E "(cache|Cache)" | wc -l

# Verify shared cache utility module exists
find apps/remix/app/utils -name "*cache*.ts" -o -name "*cache*.server.ts" | wc -l

# Verify no authentication tokens in cache keys
grep -r "token\|credential\|secret\|password" apps/remix/app/utils/cache.server.ts 2>/dev/null | grep -v "^#" | wc -l
```

**Accept when:**
- At least 80% of authenticated route handlers in the identified files implement cache layer for data retrieval operations
- A shared cache utility module exists and is imported by multiple route handlers
- Cache invalidation logic is present in mutation operations for cached resources
- Cache keys include scope identifiers (orgUrl, teamUrl, userId) to ensure tenant isolation
- No authentication tokens, credentials, or secrets are stored in cache layer
- Circuit breaker pattern is implemented around cache access with fallback to direct database queries
- Cache hit/miss metrics are tracked in application monitoring
- All cached routes have documented TTL values and invalidation triggers

<enforcement>
Claude Code MUST NOT skip or defer verification of these rules. All new authenticated route handlers MUST implement cache layer according to R-CACHE-002 through R-CACHE-009. Violations of R-CACHE-001 (sensitive tokens in cache) are critical security issues and MUST trigger immediate investigation. Cache-related bugs in production MUST be treated as high-priority issues.
</enforcement>
