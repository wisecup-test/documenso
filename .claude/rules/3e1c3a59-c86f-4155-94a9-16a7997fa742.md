<rule_activation id="3e1c3a59-c86f-4155-94a9-16a7997fa742" title="Adopt Server-Only Module Pattern for Backend-Specific Library Code: Server Only Modules" applies_to="**/*">
These rules are ALWAYS ACTIVE for all files in the codebase to enforce server-only module isolation and prevent accidental exposure of backend-specific code in client-side bundles.
</rule_activation>

### Rules

- **R-SOM-001** MAY: Server-only modules MAY export TypeScript types that can be safely imported by client code for type checking purposes.
- **R-SOM-002** MUST: All database access layers and ORM queries MUST be placed in server-only modules.
- **R-SOM-003** MUST: Authentication and authorization logic including token validation MUST be placed in server-only modules.
- **R-SOM-004** MUST: PDF generation and document processing utilities MUST be placed in server-only modules.
- **R-SOM-005** MUST: File system operations and server-side I/O MUST be placed in server-only modules.
- **R-SOM-006** MUST: API route handlers that access server-only resources MUST be placed in server-only modules.
- **R-SOM-007** MUST: Server-side configuration and environment variable access MUST be placed in server-only modules.
- **R-SOM-008** MUST: All files in the server-only directory MUST import the 'server-only' package at the top of the file to enable runtime enforcement.
- **R-SOM-009** SHOULD: Server-only code SHOULD be organized under a consistent path structure like 'packages/lib/server-only/{domain}/' where domain represents functional areas.
- **R-SOM-010** SHOULD: TypeScript path aliases SHOULD be configured to make server-only imports explicit and easily identifiable in code reviews.
- **R-SOM-011** SHOULD: ESLint rules or custom linting SHOULD be added to detect and prevent imports of server-only paths in client-side code.

### Verify

```bash
# Count server-only imports in server-only directory
grep -r "from.*server-only" packages/lib/server-only/ | wc -l

# Find server-only files missing 'server-only' package import
find packages/lib/server-only -name '*.ts' -exec grep -L "'server-only'" {} \;

# Verify client build contains no server-only references
npm run build:client && ! grep -r "server-only" dist/client/
```

**Accept when:**
- All files in the server-only directory import the 'server-only' package at the top of the file
- Client-side build output contains no references to server-only module paths or code
- Attempting to import a server-only module in a client component results in a clear error message during development or build
- No server-only code is detected in client bundles during CI/CD verification

<enforcement>
Claude Code MUST NOT skip or defer verification. All server-only module patterns MUST be enforced through automated ESLint rules, CI/CD pipeline build verification, code review checklists, and runtime errors from the 'server-only' package. Violations result in build failures, runtime errors, pull request blocks, and security review triggers.
</enforcement>
