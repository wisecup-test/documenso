<rule_activation id="431bc6eb-065f-427b-9d25-0ccf79f52f16" title="Standardize Structured Logging with Context Propagation for API and Server Operations: Endpoints Implement Structured" applies_to="**/*">
These rules are ALWAYS ACTIVE for all API endpoints, server-only functions, and service clients. All new code MUST comply with these logging standards.
</rule_activation>

### Rules

- **R-LOG-001** MUST: All API endpoints MUST implement structured logging with request context including request ID, method, path, and timestamp.
- **R-LOG-002** MUST: All server-only business logic functions including organization management, user operations, and license validation MUST implement structured logging.
- **R-LOG-003** MUST: Service clients for external API integrations and inter-service communication MUST implement structured logging.
- **R-LOG-004** MUST: Background jobs and scheduled tasks that perform business operations MUST implement structured logging.
- **R-LOG-005** MUST: Error handling and exception logging across all application layers MUST use structured logging format.
- **R-LOG-006** SHOULD: Use a shared logging utility library that encapsulates structured logging format, context propagation, and sensitive data redaction.
- **R-LOG-007** SHOULD: Establish logging level conventions: ERROR for failures requiring intervention, WARN for degraded states, INFO for significant business events, DEBUG for detailed troubleshooting.
- **R-LOG-008** SHOULD: Integrate correlation ID generation at API gateway/entry points and ensure it propagates through all downstream calls via request context or headers.
- **R-LOG-009** MUST: Do not log sensitive data including passwords, tokens, API keys, or personally identifiable information.
- **R-LOG-010** MUST: Production code MUST NOT use console.log or console.error; use structured logger instead.

### Verify

```bash
# Check for console.log/console.error usage in production code
grep -r 'console.log\|console.error' packages/ --include='*.ts' --include='*.js' | wc -l
# Expected: 0 or minimal - use structured logger instead

# Verify widespread adoption of structured logging
grep -r 'logger\.(info|error|warn|debug)' packages/api packages/lib/server-only --include='*.ts' | wc -l
# Expected: Should show widespread adoption

# Check for sensitive data patterns near logging statements
grep -r 'password\|token\|apiKey' packages/ --include='*.ts' -A 2 -B 2 | grep -i 'log' | wc -l
# Expected: 0 - no sensitive data in logs
```

**Accept when:**
- All API endpoints in packages/api contain structured logging calls with request context at entry and exit points
- Server-only functions in packages/lib/server-only log operation start/completion with entity identifiers for business operations
- No instances of console.log/console.error in production code paths (verified by linting rules)
- Automated security scanning detects no sensitive data patterns in log statements
- Log aggregation dashboard shows consistent structured log format across all services with parseable JSON
- Correlation IDs are propagated through all downstream calls and visible in log output
- Logging utility library is used consistently across all in-scope modules
