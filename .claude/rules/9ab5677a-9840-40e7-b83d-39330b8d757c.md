<rule_activation id="9ab5677a-9840-40e7-b83d-39330b8d757c" title="Enforce Server-Side Environment Variable Validation with Zod Schemas: Configuration Validation Include" applies_to="**/*">
These rules are ALWAYS ACTIVE for all server-side code that accesses environment variables or configuration values. They apply to API routes, server-only utilities, background jobs, and any code executing in Node.js runtime contexts.
</rule_activation>

### Rules

- **R-ENV-001** MAY: Configuration validation MAY include transformation logic (e.g., parsing URLs, converting strings to numbers) as part of the schema definition.
- **R-ENV-002** MUST: All server-side TypeScript/JavaScript code in Node.js runtime MUST validate environment variables and configuration using Zod schemas at module initialization.
- **R-ENV-003** MUST: API routes and endpoints that process requests MUST use validated configuration objects rather than accessing process.env directly.
- **R-ENV-004** MUST: Server-only utility modules (PDF generation, field manipulation, etc.) MUST import 'server-only' package and use Zod-validated configuration.
- **R-ENV-005** MUST: Authentication and authorization modules (SSO, groups, permissions) MUST validate all configuration inputs through Zod schemas before use.
- **R-ENV-006** MUST: Database connection and ORM configuration MUST be validated with Zod schemas at startup.
- **R-ENV-007** MUST: Third-party service integrations requiring API keys or credentials MUST validate configuration through Zod schemas.
- **R-ENV-008** MUST: Application startup MUST fail fast with clear error messages when required environment variables are missing or invalid.
- **R-ENV-009** SHOULD: Create a centralized configuration module (e.g., config/env.ts) that exports validated configuration objects using Zod schemas.
- **R-ENV-010** SHOULD: Use z.object() to define configuration schemas with explicit types (z.string(), z.number(), z.url(), etc.) and validation rules.
- **R-ENV-011** SHOULD: Use Zod's .transform() and .refine() methods for complex validation logic like URL parsing, enum validation, or cross-field dependencies.
- **R-ENV-012** SHOULD: Configure Zod schemas to redact sensitive values in error messages to prevent API keys and secrets from being logged.
- **R-ENV-013** SHOULD: Maintain .env.example files with comments that reference the Zod schema to keep documentation in sync with validation rules.

### Verify

```bash
# Check for direct process.env access in server-side code (excluding validated configuration modules)
grep -r "process\.env" --include="*.ts" --include="*.tsx" --exclude-dir=node_modules | grep -v "z\.object\|zod\|schema" | wc -l

# Find server-only modules without Zod imports
find . -name "*.ts" -path "*/server-only/*" -exec grep -L "import.*zod\|from.*zod" {} \; | wc -l

# Run configuration and environment tests with coverage
npm test -- --testPathPattern="config|env" --coverage
```

**Accept when:**
- Grep for direct process.env access returns 0 instances in server-side code (excluding validated configuration modules)
- All server-only modules import and use validated configuration objects rather than accessing environment variables directly
- Configuration validation tests achieve 100% coverage of all environment variables used in production code
- Application startup fails fast with clear error messages when required environment variables are missing or invalid
- ESLint detects no violations of direct process.env access in non-configuration modules
- All new environment variable usage includes corresponding Zod validation schemas
