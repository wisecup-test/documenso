These rules are ALWAYS ACTIVE for all E2E API test files in the packages/app-tests/e2e/api directory across all API versions (v1, v2, tRPC).

### Rules

- **R-API-001** MAY: API tests MAY include performance benchmarks and response time assertions for critical rendering paths.
- **R-API-002** MUST: All REST API endpoints (v1, v2) consumed by frontend components require E2E test coverage.
- **R-API-003** MUST: All tRPC procedures used for frontend-backend communication require E2E test coverage.
- **R-API-004** MUST: Test files MUST follow naming conventions and include both success and failure scenario coverage.
- **R-API-005** MUST: Each API endpoint exposed to frontend MUST have at least one corresponding .spec.ts file with test cases.
- **R-API-006** SHOULD: Establish shared test utilities and fixtures in a common directory to reduce duplication across test files.
- **R-API-007** SHOULD: Document the expected test structure and coverage requirements in the team's testing guidelines.

### Verify

```bash
# Count E2E API test files
find packages/app-tests/e2e/api -name '*.spec.ts' | wc -l | grep -E '^[0-9]+$'

# Count test cases across all E2E API tests
grep -r 'describe\|test\|it' packages/app-tests/e2e/api --include='*.spec.ts' | wc -l

# Verify API version directories exist
test -d packages/app-tests/e2e/api/v1 && test -d packages/app-tests/e2e/api/v2 && test -d packages/app-tests/e2e/api/trpc
```

**Accept when:**
- The packages/app-tests/e2e/api directory contains test specifications organized by API version (v1, v2, trpc)
- Each API endpoint exposed to frontend has at least one corresponding .spec.ts file with test cases
- Test files follow naming conventions and include both success and failure scenario coverage
- CI pipeline successfully executes all E2E API tests and reports coverage metrics
- Performance benchmarks and response time assertions are included for critical rendering paths where applicable
