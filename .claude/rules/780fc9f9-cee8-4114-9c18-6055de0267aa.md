<rule_activation id="780fc9f9-cee8-4114-9c18-6055de0267aa" title="Adopt Server-Only Data Access Layer with Function-Based Repository Pattern: Client Side Code" applies_to="**/*">
These rules are ALWAYS ACTIVE for all server-side data access operations. All database queries and data mutations MUST be implemented through the server-only data access layer.
</rule_activation>

### Rules

- **R-SODAL-001** MUST_NOT: Client-side code MUST NOT directly import or execute database queries; all data access MUST go through the server-only layer
- **R-SODAL-002** MUST: All database queries and mutations for application data MUST be implemented in the server-only layer
- **R-SODAL-003** MUST: CRUD operations for domain entities (fields, folders, API tokens, etc.) MUST use the server-only data access layer
- **R-SODAL-004** MUST: Server-side API route handlers and server components MUST access data through the server-only layer
- **R-SODAL-005** MUST: Background jobs and scheduled tasks requiring data access MUST use the server-only layer
- **R-SODAL-006** MUST: Authentication and authorization checks requiring database queries MUST be implemented in the server-only layer
- **R-SODAL-007** SHOULD: Data access functions SHOULD follow naming conventions (get-*, find-*, update-*, delete-*, remove-*) and be isolated in individual files
- **R-SODAL-008** SHOULD: Data access functions SHOULD be organized by domain entity in subdirectories with clean import paths via index files
- **R-SODAL-009** MAY: Batch operations and internal variants (e.g., find-folders-internal) MAY be created to optimize multiple queries

### Verify

```bash
# Check for client-side imports of server-only modules
grep -r 'from.*server-only' --include='*.tsx' --include='*.ts' apps/*/app/**/page.tsx apps/*/app/**/layout.tsx | grep -v '.server.' || echo 'No client-side imports of server-only modules detected'

# Count data access functions following naming convention
find packages/lib/server-only -type f -name '*.ts' | xargs -I {} basename {} | grep -E '^(get|find|update|delete|remove)-.*\.ts$' | wc -l

# Verify server-only directory exists and contains exported functions
test -d packages/lib/server-only && find packages/lib/server-only -type f -name '*.ts' -exec grep -l 'export.*function' {} \; | wc -l
```

**Accept when:**
- All database access operations are located in the server-only directory structure with no direct database queries in client-side code
- Each data access function follows the naming convention (get-*, find-*, update-*, delete-*, remove-*) and is isolated in its own file
- Build process successfully prevents client-side imports of server-only modules and fails with clear error messages when violations are detected
- Code review checklist includes verification that new data access operations follow the established pattern
- No grep results show client-side imports of server-only modules in page.tsx or layout.tsx files
- Automated ESLint rules and CI pipeline checks pass without violations

<enforcement>
Claude Code MUST NOT skip or defer verification of server-only data access layer compliance. Build-time enforcement through bundler configuration (Next.js 'server-only' package) is mandatory. Violations are blocking and must be resolved before merge.
</enforcement>
