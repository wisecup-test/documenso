<rule_activation id="7775597f-e1ae-4e3d-ae5e-bf8a571adaa1" title="Adopt Server-Only Module Pattern for External API Integrations: External Integrations Implemented" applies_to="**/*">
These rules are ALWAYS ACTIVE for all files in the codebase. Server-only module patterns enforce strict isolation of external API integrations, sensitive operations, and credentials from client-side code to prevent security vulnerabilities and compliance violations.
</rule_activation>

### Rules

- **R-SOMP-001** MUST: All external API integrations MUST be implemented in modules explicitly marked as server-only and placed under designated server-only directory structures (e.g., `packages/lib/server-only/`).

- **R-SOMP-002** MUST: Server-only modules MUST include all external API integrations (third-party services, payment processors, authentication providers), PDF generation and document manipulation operations, audit log rendering and compliance reporting, recipient validation and turn-based workflow logic, database queries that expose sensitive data or business logic, and server-side rendering operations that require privileged access.

- **R-SOMP-003** MUST: Server-only modules MUST NOT be imported directly in client-side code, routes, components, or hooks.

- **R-SOMP-004** MUST: Build processes MUST fail if server-only code is detected in client JavaScript bundles.

- **R-SOMP-005** MUST: CI pipeline MUST include automated checks that fail on detection of server-only imports in client contexts.

- **R-SOMP-006** SHOULD: Use TypeScript path mapping to make server-only imports explicit (e.g., `@server/*` vs `@client/*`).

- **R-SOMP-007** SHOULD: Implement ESLint rules to detect imports from server-only paths in client-side files.

- **R-SOMP-008** MAY: Exceptions for type definitions or interfaces with no implementation logic (EXC-001) or development/testing mock implementations (EXC-002) require security team approval and documented compensating controls.

### Verify

```bash
# Check for client-side imports of server-only modules
grep -r "from.*server-only" apps/remix/app/routes --include="*.tsx" --include="*.ts" | grep -v "loader\|action" || echo "No client-side server-only imports detected"

# Verify no sensitive credentials are exported from server-only modules
find packages/lib/server-only -type f -name "*.ts" -exec grep -l "export.*API_KEY\|SECRET\|PASSWORD" {} \; | wc -l

# Check client bundle size for suspicious server-only code
npm run build && du -sh dist/client/**/*.js | awk '{if($1 ~ /M/ && $1+0 > 5) print "Warning: Large client bundle detected: " $0}'
```

**Accept when:**
- All server-only modules reside under designated server-only directory structures (e.g., `packages/lib/server-only/`) and are not imported directly in client-side code
- Build process successfully prevents bundling of server-only code in client JavaScript outputs
- CI pipeline includes automated checks that fail on detection of server-only imports in client contexts
- All external API integrations, PDF operations, and audit log rendering are implemented as server-only modules
- ESLint rules enforce server-only import restrictions and pass without violations
- Bundle analysis confirms no server-only code appears in client chunks

<enforcement>
Claude Code MUST NOT skip or defer verification of server-only module boundaries. Build-time enforcement and CI checks are mandatory before deployment. Any violations detected in production builds require immediate rollback and security team notification.
</enforcement>
