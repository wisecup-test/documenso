<rule_activation id="9ff4e19d-58f2-4a9f-96d9-960ed050d558" title="Adopt Monorepo Package Structure with Scoped Modules: Packages Published Independently" applies_to="**/*">
These rules are ALWAYS ACTIVE for all files in the monorepo to enforce consistent package structure, clear architectural boundaries, and independent package organization.
</rule_activation>

### Rules

- **R-MONO-001** MAY: Packages MAY be published independently if they provide reusable functionality across multiple projects.
- **R-MONO-002** MUST: All TypeScript/JavaScript modules intended for reuse across multiple application components MUST be organized under the `packages/` directory structure.
- **R-MONO-003** MUST: API contract definitions, OpenAPI specifications, and versioned API interfaces MUST be located in `packages/api/` with versioned subdirectories (e.g., `packages/api/v1/`).
- **R-MONO-004** MUST: Server-only utilities including authentication, authorization, and data access logic MUST be isolated in `packages/lib/server-only/` to prevent accidental client-side imports.
- **R-MONO-005** MUST: No client-side code or components MAY import from `packages/lib/server-only/`.
- **R-MONO-006** SHOULD: Use TypeScript path mapping in `tsconfig.json` to enable clean imports (e.g., `@packages/api/v1/contract` instead of relative paths).
- **R-MONO-007** SHOULD: Document the purpose and scope of each package in its own `README.md` file.
- **R-MONO-008** SHOULD: Establish naming conventions: `packages/api` for public contracts, `packages/lib` for shared utilities, `packages/app-*` for application-specific code.
- **R-MONO-009** MUST: Circular dependencies between packages are prohibited and MUST be detected and resolved.
- **R-MONO-010** SHOULD: Consider using monorepo management tools (Turborepo, Nx, or Lerna) for builds, caching, and dependency orchestration.

### Verify

```bash
# Count TypeScript files in established package structure
find packages/ -type f -name '*.ts' | grep -E 'packages/(api|lib|app-tests)' | wc -l

# Verify no client-side imports of server-only code
grep -r 'packages/lib/server-only' --include='*.ts' --exclude-dir=node_modules | grep -v 'server-only' || echo 'No client-side imports of server-only code detected'

# Verify API contract structure exists
test -d packages/api/v1 && test -f packages/api/v1/openapi.ts && test -f packages/api/v1/contract.ts && echo 'API contract structure verified'

# Verify no circular dependencies (requires dependency-cruiser or madge)
# madge --circular packages/ || echo 'Circular dependency check completed'
```

**Accept when:**
- All TypeScript files under `packages/` directory follow the established structure with clear package boundaries (api, lib, app-tests)
- No imports of `packages/lib/server-only` code exist in client-side bundles or components
- API contracts and OpenAPI specifications are located in `packages/api` with versioned subdirectories
- Automated dependency graph validation passes without circular dependency errors
- Build-time checks confirm server-only code is not included in client bundles
- Package structure review passes code review checklist
