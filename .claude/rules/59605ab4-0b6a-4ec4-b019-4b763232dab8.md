<rule_activation id="59605ab4-0b6a-4ec4-b019-4b763232dab8" title="Enforce Server-Only Module Boundaries for Sensitive Operations: Teams Use Typescript" applies_to="**/*">
These rules are ALWAYS ACTIVE for all server-side code modules handling sensitive operations, data access patterns, and environment-specific functionality. They apply to all new and existing server-only modules.
</rule_activation>

### Rules

- **R-SOOB-001** MAY: Teams MAY use TypeScript path aliases or module resolution configuration to enforce import restrictions at development time.
- **R-SOOB-002** MUST: All database access and ORM operations must be confined to server-only modules.
- **R-SOOB-003** MUST: PDF generation and document processing logic must be confined to server-only modules.
- **R-SOOB-004** MUST: Audit logging and compliance tracking must be confined to server-only modules.
- **R-SOOB-005** MUST: Field manipulation and template processing with business rules must be confined to server-only modules.
- **R-SOOB-006** MUST: Authentication and authorization enforcement logic must be confined to server-only modules.
- **R-SOOB-007** MUST: API route handlers and server-side data fetching must be confined to server-only modules.
- **R-SOOB-008** MUST: Environment variable access and configuration management must be confined to server-only modules.
- **R-SOOB-009** MUST: Third-party service integrations requiring API keys or secrets must be confined to server-only modules.
- **R-SOOB-010** MUST: Server-only modules must include 'server-only' package import or equivalent framework marker at the top of the file.
- **R-SOOB-011** SHOULD: Use directory structure conventions by placing server-only code in /server-only/, /api/, or similar directories.
- **R-SOOB-012** SHOULD: Configure ESLint to restrict imports from server-only paths in client code.
- **R-SOOB-013** MAY: Shared utility functions that are truly environment-agnostic (e.g., date formatting, string manipulation) may exist outside server-only boundaries (EXC-001).
- **R-SOOB-014** MAY: Type definitions and interfaces may be shared between client and server when they contain no implementation logic (EXC-002).

### Verify

```bash
# Check for client code importing server-only modules
grep -r "from.*server-only" apps/*/client apps/*/components --include="*.ts" --include="*.tsx" && echo "VIOLATION: Client code importing server-only modules" || echo "PASS"

# Verify client bundles do not contain server-only code
npm run build && npx bundle-analyzer --check-server-only || echo "Verify client bundles do not contain server-only code"

# Check for server-only files missing runtime guard
find . -path '*/server-only/*' -name '*.ts' -exec grep -L "'server-only'" {} \; | head -5 && echo "WARNING: Server-only files missing runtime guard" || echo "PASS"
```

**Accept when:**
- All modules in server-only directories include 'server-only' package import or equivalent framework marker
- Build process successfully excludes server-only modules from client bundles (verified via bundle analysis)
- No client-side code directly imports from server-only paths (verified via static analysis or linting)
- CI pipeline includes automated checks that fail on server-only boundary violations
- ESLint rules with import restrictions are configured for client-side directories
- Pre-commit hooks or CI checks scan for common violations

Claude Code MUST verify server-only module boundaries are enforced via automated CI checks, bundle analysis, and ESLint rules before approving code changes. Violations MUST block PR merges. Runtime errors via 'server-only' package MUST be caught in development and staging environments.
</enforcement>
