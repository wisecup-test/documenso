<rule_activation id="1ebb7fab-1894-4a1c-9eda-00dd134909c9" title="Standardize Logging Configuration for Environment-Aware Runtime Observability: Structured Logging Formats" applies_to="**/*">
These rules are ALWAYS ACTIVE for all runtime components that require logging and observability capabilities.
</rule_activation>

### Rules

- **R-LOG-001** SHOULD: Structured logging formats (JSON) SHOULD be preferred in production environments for machine parsing.

### Verify

```bash
# Check for environment variable references for logging configuration
grep -r 'process\.env.*LOG' --include='*.ts' --include='*.tsx' --include='*.js' | wc -l

# Check for logger initialization or configuration code
grep -r 'logger\.configure\|createLogger\|winston\|pino' --include='*.ts' --include='*.tsx' | head -20

# Check for environment template files documenting logging configuration
find . -name '*.env.example' -o -name '*.env.template' | xargs grep -l 'LOG_LEVEL\|LOG_FORMAT' 2>/dev/null
```

**Accept when:**
- Environment variable references for logging configuration are found in multiple source files
- Logger initialization or configuration code is present in the codebase
- Environment template files document logging-related configuration variables

<enforcement>
Claude Code MUST verify logging configuration is environment-aware and MUST NOT skip verification. Code review checklist MUST include verification of environment-based logging configuration. Automated linting rules MUST detect hardcoded log levels. CI pipeline MUST include tests verifying logging configuration responds to environment variables. Violations MUST be treated as blocking issues with exception process requiring engineering lead approval.
</enforcement>
