<rule_activation id="03547363-d3fd-4ee4-8bfe-25378a377e1c" title="Adopt Server-Only Module Pattern for External API Integrations: Build Tooling Enforce" applies_to="**/*">
These rules are ALWAYS ACTIVE for all files matching `**/*`. Server-only modules handling external API integrations, PDF generation, and sensitive operations must never execute in client-side contexts. Build tooling MUST enforce server-only module boundaries and prevent client-side bundling of server-only code.
</rule_activation>

### Rules

- **R-SERVERONLY-001** MUST: Build tooling MUST enforce server-only module boundaries and prevent client-side bundling of server-only code.
- **R-SERVERONLY-002** MUST: All external API integrations (third-party services, payment processors, authentication providers) MUST be implemented as server-only modules.
- **R-SERVERONLY-003** MUST: PDF generation and document manipulation operations MUST be isolated in server-only modules.
- **R-SERVERONLY-004** MUST: Audit log rendering and compliance reporting MUST be implemented as server-only modules.
- **R-SERVERONLY-005** MUST: Recipient validation and turn-based workflow logic MUST be server-only.
- **R-SERVERONLY-006** MUST: Database queries that expose sensitive data or business logic MUST be server-only.
- **R-SERVERONLY-007** MUST: Server-side rendering operations requiring privileged access MUST be server-only.
- **R-SERVERONLY-008** MUST: All server-only modules MUST reside under designated `packages/lib/server-only/` directory structure, organized by domain (pdf, recipient, auth, etc.).
- **R-SERVERONLY-009** MUST: ESLint rules MUST detect and prevent imports from server-only paths in client-side files (routes, components, hooks).
- **R-SERVERONLY-010** MUST: CI pipeline MUST include automated checks that fail on detection of server-only imports in client contexts.
- **R-SERVERONLY-011** MUST: Build process MUST fail if server-only paths are detected in client bundle analysis.
- **R-SERVERONLY-012** SHOULD: Use TypeScript path mapping to make server-only imports explicit (e.g., '@server/*' vs '@client/*').
- **R-SERVERONLY-013** SHOULD: Document framework-specific patterns for exposing server-only functionality (Remix loaders/actions, Next.js API routes, tRPC procedures).

### Verify

```bash
# Check for client-side imports of server-only modules in Remix routes
grep -r "from.*server-only" apps/remix/app/routes --include="*.tsx" --include="*.ts" | grep -v "loader\|action" || echo "No client-side server-only imports detected"

# Count server-only modules exporting sensitive credentials
find packages/lib/server-only -type f -name "*.ts" -exec grep -l "export.*API_KEY\|SECRET\|PASSWORD" {} \; | wc -l

# Verify client bundle size and check for server-only code leakage
npm run build && du -sh dist/client/**/*.js | awk '{if($1 ~ /M/ && $1+0 > 5) print "Warning: Large client bundle detected: " $0}'
```

**Accept when:**
- All server-only modules reside under designated server-only directory structures and are not imported directly in client-side code
- Build process successfully prevents bundling of server-only code in client JavaScript outputs
- CI pipeline includes automated checks that fail on detection of server-only imports in client contexts
- All external API integrations, PDF operations, and audit log rendering are implemented as server-only modules
- No server-only imports are detected in client-side routes, components, or hooks (except within loader/action functions)
- ESLint validation passes with no violations of server-only import restrictions
- Bundle analysis confirms no server-only code appears in client chunks

<enforcement>
Claude Code MUST NOT skip or defer verification. All rules in this activation are mandatory. Build failures blocking deployment when server-only code is detected in client bundles are non-negotiable. ESLint violations require resolution before PR approval. Security team notification is required for any violations detected in production builds.
</enforcement>
