<rule_activation id="a40df72a-d9d7-4ba9-bbaa-36e746169edd" title="Adopt E2E API Testing for Frontend Rendering Model Validation: E2e Tests Cover" applies_to="**/*">
These rules are ALWAYS ACTIVE for all files matching API endpoints, test specifications, and frontend rendering integrations.
</rule_activation>

### Rules

- **R-E2E-001** SHOULD: E2E tests SHOULD cover critical user workflows including document operations, template management, envelope distribution, and search functionality.
- **R-E2E-002** MUST: All REST API endpoints (v1, v2) that serve frontend applications MUST have E2E test coverage.
- **R-E2E-003** MUST: All tRPC procedures that provide data for UI rendering MUST have E2E test coverage.
- **R-E2E-004** MUST: API endpoints for document search, retrieval, and management MUST be covered by E2E tests.
- **R-E2E-005** MUST: API endpoints for template operations and field prefilling MUST be covered by E2E tests.
- **R-E2E-006** MUST: API endpoints for envelope distribution and validation MUST be covered by E2E tests.
- **R-E2E-007** MUST: API endpoints for embedding and presigning operations MUST be covered by E2E tests.
- **R-E2E-008** MUST: E2E test files MUST be organized in directory structure matching API versioning: e2e/api/v1/, e2e/api/v2/, e2e/api/trpc/.
- **R-E2E-009** SHOULD: Test names SHOULD be descriptive and clearly indicate the API endpoint and scenario being tested (e.g., 'search-documents.spec.ts', 'template-field-prefill.spec.ts').
- **R-E2E-010** SHOULD: Shared test utilities SHOULD be created for common operations like authentication, test data setup, and response validation.
- **R-E2E-011** SHOULD: Test data factories SHOULD be implemented to generate consistent, realistic test data for API requests.
- **R-E2E-012** MAY: API client libraries MAY be used in tests to ensure tests exercise the same code paths as production frontend applications.

### Exceptions

- **EXC-001**: API endpoints are deprecated and scheduled for removal within 30 days.
- **EXC-002**: Internal admin APIs with no direct frontend consumption.

### Verify

```bash
# Verify at least 8 E2E API test specification files exist
find packages/app-tests/e2e/api -name '*.spec.ts' | wc -l | grep -E '^[8-9]|^[1-9][0-9]+$'

# Count test cases across all E2E API tests
grep -r 'describe\|test\|it' packages/app-tests/e2e/api --include='*.spec.ts' | wc -l

# List all E2E API test files
npm test -- --testPathPattern=e2e/api --listTests | wc -l
```

**Accept when:**
- At least 8 E2E API test specification files exist covering major API endpoints
- All API versions (v1, v2, tRPC) have corresponding test coverage for frontend-facing endpoints
- E2E test suite runs successfully in CI/CD pipeline with >90% pass rate
- Test coverage includes document operations, template management, envelope workflows, and search functionality

<enforcement>
Claude Code MUST verify E2E API test coverage on every pull request. Pull requests without corresponding E2E tests for API changes MUST be blocked from merging. CI/CD pipeline MUST fail if E2E test suite does not pass. Test coverage regression MUST trigger automated alerts to engineering team.
</enforcement>
