<rule_activation id="d66f0443-c5a3-4938-afa2-462c31b38503" title="Adopt Server-Only Data Access Layer with Transactional Boundaries: Data Mutation Operations" applies_to="**/*">
These rules are ALWAYS ACTIVE for all server-side data access operations. All data mutations and queries that interact with the database MUST follow the patterns defined herein.
</rule_activation>

### Rules

- **R-SODAL-001** MUST: Data mutation operations (create, update, delete) MUST be encapsulated in dedicated service functions with clear transactional boundaries.
- **R-SODAL-002** MUST: All database queries and mutations across the application MUST be executed server-side with proper transactional semantics and security controls.
- **R-SODAL-003** MUST: Server-side API endpoints that interact with persistent storage MUST implement authentication and authorization checks for data access.
- **R-SODAL-004** MUST: Data validation and transformation logic MUST occur before persistence in the server-only layer.
- **R-SODAL-005** SHOULD: Organize server-only data access functions by domain entity in a clear directory structure (e.g., server-only/folder/, server-only/recipient/, server-only/profile/).
- **R-SODAL-006** SHOULD: Use TypeScript to define clear input/output types for each data access function, enabling compile-time validation and better IDE support.
- **R-SODAL-007** SHOULD: Implement a consistent error handling strategy across all server-only functions, using typed exceptions or result types to communicate failures.
- **R-SODAL-008** MAY: Read-only database queries for public data that requires no authentication may be executed in edge functions or middleware for performance optimization (EX-001).
- **R-SODAL-009** MAY: Database migrations and schema management tools may access the database directly outside the server-only layer (EX-002).

### Verify

```bash
# Verify no client-side database imports detected
grep -r 'import.*prisma\|import.*db' apps/*/app --exclude-dir=node_modules | grep -v 'server-only' | grep -v '.server.' || echo 'No client-side database imports detected'

# Verify at least 5 server-only modules exist
find packages/lib/server-only -type f -name '*.ts' | wc -l | awk '{if($1 >= 5) print "PASS: Found " $1 " server-only modules"; else print "FAIL: Expected at least 5 server-only modules"}'

# Verify at least 3 data mutation functions exist
grep -r 'export.*function.*delete\|export.*function.*create\|export.*function.*update' packages/lib/server-only --include='*.ts' | wc -l | awk '{if($1 >= 3) print "PASS: Found " $1 " data mutation functions"; else print "FAIL: Expected at least 3 mutation functions"}'
```

**Accept when:**
- No client-side code directly imports database clients or executes queries; all database access is mediated through server-only layer
- At least 5 server-only data access modules exist, organized by domain entity, each handling specific business operations
- All data mutation operations (create, update, delete) are implemented as exported functions in the server-only layer with proper error handling
- No database client imports detected outside the server-only directory structure
- All data access functions include preconditions, postconditions, and side effects documentation

<enforcement>
Claude Code MUST NOT skip or defer verification. All data access patterns MUST be verified against the server-only layer requirements before accepting code changes. Build failures for linting violations that detect client-side database access are mandatory.
</enforcement>
