<rule_activation id="1c992304-a731-44f0-9b3e-e279e6a457e0" title="Adopt Server-Only Module Pattern for External API Integrations: Credentials Secrets Sensitive" applies_to="**/*">
These rules are ALWAYS ACTIVE for all files in the codebase to enforce strict isolation of server-only modules containing API credentials, secrets, and sensitive operations from client-side code.
</rule_activation>

### Rules

- **R-SERVERONLY-001** MUST: API credentials, secrets, and sensitive configuration MUST only be accessed within server-only module boundaries.
- **R-SERVERONLY-002** MUST: All external API integrations (third-party services, payment processors, authentication providers) MUST be implemented as server-only modules.
- **R-SERVERONLY-003** MUST: PDF generation and document manipulation operations MUST be isolated to server-only modules.
- **R-SERVERONLY-004** MUST: Audit log rendering and compliance reporting MUST be implemented as server-only modules.
- **R-SERVERONLY-005** MUST: Recipient validation and turn-based workflow logic MUST be isolated to server-only modules.
- **R-SERVERONLY-006** MUST: Database queries that expose sensitive data or business logic MUST be isolated to server-only modules.
- **R-SERVERONLY-007** MUST: Server-side rendering operations that require privileged access MUST be isolated to server-only modules.
- **R-SERVERONLY-008** MUST: Build process MUST fail if server-only code is detected in client JavaScript bundles.
- **R-SERVERONLY-009** MUST: ESLint rules MUST enforce that server-only modules are not imported in client-side files (routes, components, hooks).
- **R-SERVERONLY-010** SHOULD: Use TypeScript path mapping to make server-only imports explicit (e.g., '@server/*' vs '@client/*').
- **R-SERVERONLY-011** SHOULD: Organize server-only modules under 'packages/lib/server-only/' directory structure, organized by domain (pdf, recipient, auth, etc.).
- **R-SERVERONLY-012** MAY: Client-side API calls to internal endpoints are out of scope (handled by framework routing).
- **R-SERVERONLY-013** MAY: Public-facing REST or GraphQL endpoints are out of scope (covered by separate API design patterns).
- **R-SERVERONLY-014** MAY: Shared utility functions containing no sensitive logic or credentials are out of scope.
- **R-SERVERONLY-015** MAY: Type definitions and interface contracts shared between client and server are out of scope.

### Exceptions

- **EXC-001**: A module contains only type definitions or interfaces with no implementation logic.
- **EXC-002**: Development/testing environments require mock implementations for client-side testing.

### Verify

```bash
# Check for client-side imports of server-only modules in Remix routes
grep -r "from.*server-only" apps/remix/app/routes --include="*.tsx" --include="*.ts" | grep -v "loader\|action" || echo "No client-side server-only imports detected"

# Count server-only modules exporting credentials or secrets
find packages/lib/server-only -type f -name "*.ts" -exec grep -l "export.*API_KEY\|SECRET\|PASSWORD" {} \; | wc -l

# Verify client bundle size and check for server-only code
npm run build && du -sh dist/client/**/*.js | awk '{if($1 ~ /M/ && $1+0 > 5) print "Warning: Large client bundle detected: " $0}'
```

**Accept when:**
- All server-only modules reside under designated server-only directory structures and are not imported directly in client-side code.
- Build process successfully prevents bundling of server-only code in client JavaScript outputs.
- CI pipeline includes automated checks that fail on detection of server-only imports in client contexts.
- All external API integrations, PDF operations, and audit log rendering are implemented as server-only modules.
- ESLint rules pass with no violations of server-only import restrictions.
- No server-only code is detected in client bundle analysis.

<enforcement>
Claude Code MUST NOT skip or defer verification of server-only module boundaries. All R-SERVERONLY rules marked MUST are mandatory and violations MUST block deployment. Build failures on server-only code detection in client bundles are non-negotiable. Security team notification is required for any violations detected in production builds.
</enforcement>
