<rule_activation id="b9ccbfc4-493a-4840-b3ec-12f2196724f9" title="Standardize Environment Variable Configuration Management Pattern: Configuration Schemas Documented" applies_to="**/*">
These rules are ALWAYS ACTIVE for all runtime configuration and environment management implementations across server-side application code, API implementations, service integrations, PDF processing utilities, database connections, feature flags, and file system paths.
</rule_activation>

### Rules

- **R-CONFIG-001** SHOULD: Configuration schemas SHOULD be documented and version-controlled to track changes in configuration requirements over time.

### Verify

```bash
# Check for direct process.env access outside configuration modules
grep -r 'process\.env\.' --include='*.ts' --include='*.js' packages/ | grep -v 'config\|environment' | wc -l

# Verify centralized configuration modules exist
find packages/ -name 'config.ts' -o -name 'environment.ts' -o -name 'env.ts' | head -5

# Check for configuration imports in API and library code
grep -r 'import.*config.*from' --include='*.ts' packages/api packages/lib | wc -l
```

**Accept when:**
- Direct process.env access is minimal or zero outside of dedicated configuration modules
- At least one centralized configuration module exists that exports typed configuration objects
- Configuration imports are present in API and library code indicating usage of centralized configuration
- Configuration schemas are documented with .env.example files or dedicated configuration documentation
- Configuration loading occurs at application entry point with validation and clear error messages

<enforcement>
Claude Code MUST verify configuration patterns through code review, ESLint rules, and CI pipeline checks. Pull requests with direct environment variable access outside configuration modules MUST be flagged. Linting violations MUST be resolved before merge to main branches. Existing violations MUST be tracked as technical debt with prioritized remediation.
</enforcement>
