<rule_activation id="f21f8714-3028-404c-9b4e-edd7a203a435" title="Adopt Monorepo Package Structure with Scoped Modules: Each Package Have" applies_to="**/*">
These rules are ALWAYS ACTIVE for all files in the monorepo to enforce consistent package structure, clear architectural boundaries, and proper code organization across scoped modules.
</rule_activation>

### Rules

- **R-MONO-001** SHOULD: Each package SHOULD have a clear single responsibility (API contracts, server utilities, test fixtures, etc.)
- **R-MONO-002** MUST: All TypeScript/JavaScript modules intended for reuse across multiple application components MUST be organized under the packages/ directory structure
- **R-MONO-003** MUST: API contract definitions, OpenAPI specifications, and versioned API interfaces MUST be located in packages/api with versioned subdirectories
- **R-MONO-004** MUST: Server-only utilities including authentication, authorization, and data access logic MUST be isolated in packages/lib/server-only to prevent client-side imports
- **R-MONO-005** MUST: Test constants, fixtures, and shared testing utilities MUST be organized in dedicated test packages (e.g., packages/app-tests)
- **R-MONO-006** MUST: No imports of packages/lib/server-only code MUST exist in client-side bundles or components
- **R-MONO-007** SHOULD: Use TypeScript path mapping in tsconfig.json to enable clean imports (e.g., '@packages/api/v1/contract' instead of relative paths)
- **R-MONO-008** SHOULD: Establish naming conventions for packages: packages/api for public contracts, packages/lib for shared utilities, packages/app-* for application-specific code
- **R-MONO-009** MAY: Consider using tools like Turborepo, Nx, or Lerna to manage monorepo builds, caching, and dependency orchestration
- **R-MONO-010** SHOULD: Document the purpose and scope of each package in its own README.md file to guide developers

### Verify

```bash
# Count TypeScript files in expected package structure
find packages/ -type f -name '*.ts' | grep -E 'packages/(api|lib|app-tests)' | wc -l

# Verify no client-side imports of server-only code
grep -r 'packages/lib/server-only' --include='*.ts' --exclude-dir=node_modules | grep -v 'server-only' || echo 'No client-side imports of server-only code'

# Verify API contract structure exists
test -d packages/api/v1 && test -f packages/api/v1/openapi.ts && test -f packages/api/v1/contract.ts

# Validate no circular dependencies
madge --circular packages/ || echo 'Dependency graph validation required'
```

**Accept when:**
- All TypeScript files under packages/ directory follow the established structure with clear package boundaries (api, lib, app-tests)
- No imports of packages/lib/server-only code exist in client-side bundles or components
- API contracts and OpenAPI specifications are located in packages/api with versioned subdirectories
- Automated dependency graph validation passes without circular dependency errors
- Each package has a documented single responsibility
- TypeScript path mappings are configured for clean imports across packages
