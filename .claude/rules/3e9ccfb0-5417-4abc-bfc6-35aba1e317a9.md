<rule_activation id="3e9ccfb0-5417-4abc-bfc6-35aba1e317a9" title="Adopt Server-Only Data Access Layer with Function-Based Repository Pattern: Internal Variants Data" applies_to="**/*">
These rules are ALWAYS ACTIVE for all server-side data access operations. All database queries and data mutations MUST be implemented through the server-only data access layer.
</rule_activation>

### Rules

- **R-SODA-001** MUST: Implement all database queries and mutations through the server-only data access layer, never directly in client-side code.
- **R-SODA-002** MUST: Organize data access functions by domain entity in subdirectories (field/, folder/, etc.) within the server-only directory structure.
- **R-SODA-003** MUST: Follow naming conventions for data access functions: get-*, find-*, update-*, delete-*, remove-* patterns, each isolated in its own file.
- **R-SODA-004** MUST: Use module bundler configuration (e.g., Next.js 'server-only' package) to enforce server-side execution and fail builds if server-only modules are imported in client code.
- **R-SODA-005** MUST: Prevent client-side imports of server-only modules through build-time enforcement and ESLint rules.
- **R-SODA-006** MAY: Create internal variants of data access functions (e.g., find-folders-internal) for operations requiring different authorization contexts or additional internal parameters.
- **R-SODA-007** SHOULD: Implement consistent error handling and TypeScript typing across all data access functions.
- **R-SODA-008** SHOULD: Use a base query builder or database client wrapper that all data access functions use to ensure consistent connection handling and query logging.
- **R-SODA-009** SHOULD: Document common patterns for each operation type (get, find, update, delete) including parameter validation, error handling, and return type conventions.

### Verify

```bash
# Check for client-side imports of server-only modules
grep -r 'from.*server-only' --include='*.tsx' --include='*.ts' apps/*/app/**/page.tsx apps/*/app/**/layout.tsx | grep -v '.server.' || echo 'No client-side imports of server-only modules detected'

# Count data access functions following naming convention
find packages/lib/server-only -type f -name '*.ts' | xargs -I {} basename {} | grep -E '^(get|find|update|delete|remove)-.*\.ts$' | wc -l

# Verify server-only directory exists and contains exported functions
test -d packages/lib/server-only && find packages/lib/server-only -type f -name '*.ts' -exec grep -l 'export.*function' {} \; | wc -l
```

**Accept when:**
- All database access operations are located in the server-only directory structure with no direct database queries in client-side code
- Each data access function follows the naming convention (get-*, find-*, update-*, delete-*, remove-*) and is isolated in its own file
- Build process successfully prevents client-side imports of server-only modules and fails with clear error messages when violations are detected
- Code review checklist includes verification that new data access operations follow the established pattern
- No client-side page or layout files import from server-only modules

<enforcement>
Claude Code MUST NOT skip or defer verification. All database access operations MUST comply with the server-only data access layer pattern. Build failures for client-side imports of server-only modules are blocking violations.
</enforcement>
