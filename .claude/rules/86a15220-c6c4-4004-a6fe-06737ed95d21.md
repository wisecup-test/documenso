These rules are ALWAYS ACTIVE for all service boundary definitions and configuration management patterns across the codebase.
</rule_activation>

### Rules

- **R-SBD-001** SHOULD: Configuration schemas SHOULD be validated at application startup to fail fast on misconfiguration.

### Verify

```bash
# Count environment variable usage for service boundaries
grep -r "process.env" --include="*.ts" --include="*.tsx" | grep -E "(URL|ENDPOINT|API)" | wc -l

# Find configuration files and examples
find . -name "*.env.example" -o -name "config.ts" -o -name "services.config.ts" | wc -l

# Detect environment-specific conditionals with hardcoded URLs
grep -r "if.*NODE_ENV" --include="*.ts" --include="*.tsx" | grep -E "(http|https|://)" | wc -l
```

**Accept when:**
- All service boundary URLs and endpoints are loaded from environment variables or configuration files, not hardcoded strings
- Configuration validation exists at application startup that fails fast on missing or invalid service definitions
- No environment-specific conditionals (if NODE_ENV) contain hardcoded service URLs or endpoints
- Documentation exists listing all required environment variables for service boundary configuration

<enforcement>
Claude Code MUST verify all rules in this activation block. Configuration validation at startup is mandatory for all service boundary definitions. Violations detected during code review or CI pipeline checks MUST result in build failure or required remediation.
</enforcement>
