<rule_activation id="92196daf-f89a-44dd-9c1d-e125237f3e73" title="Adopt Server-Only Module Pattern for External API Integrations: Pdf Generation Document" applies_to="**/*">
These rules are ALWAYS ACTIVE for all files in the codebase to enforce strict isolation of server-only modules handling external API integrations, PDF generation, and sensitive operations from client-side code.
</rule_activation>

### Rules

- **R-SERVERONLY-001** MUST: PDF generation, document manipulation, and audit log rendering operations MUST be implemented as server-only modules.
- **R-SERVERONLY-002** MUST: All external API integrations (third-party services, payment processors, authentication providers) MUST be implemented as server-only modules.
- **R-SERVERONLY-003** MUST: Recipient validation and turn-based workflow logic MUST be implemented as server-only modules.
- **R-SERVERONLY-004** MUST: Database queries that expose sensitive data or business logic MUST be implemented as server-only modules.
- **R-SERVERONLY-005** MUST: Server-side rendering operations that require privileged access MUST be implemented as server-only modules.
- **R-SERVERONLY-006** MUST: All server-only modules MUST reside under designated server-only directory structures (e.g., `packages/lib/server-only/`) and MUST NOT be imported directly in client-side code.
- **R-SERVERONLY-007** MUST: Build process MUST prevent bundling of server-only code in client JavaScript outputs.
- **R-SERVERONLY-008** MUST: CI pipeline MUST include automated checks that fail on detection of server-only imports in client contexts.
- **R-SERVERONLY-009** SHOULD: Use TypeScript path mapping to make server-only imports explicit (e.g., `@server/*` vs `@client/*`).
- **R-SERVERONLY-010** SHOULD: Configure build tools (Webpack, Vite, etc.) to fail builds if server-only paths are detected in client bundle analysis.
- **R-SERVERONLY-011** SHOULD: Implement ESLint plugin or rule to detect imports from server-only paths in client-side files (routes, components, hooks).

### Verify

```bash
# Check for client-side imports of server-only modules
grep -r "from.*server-only" apps/remix/app/routes --include="*.tsx" --include="*.ts" | grep -v "loader\|action" || echo "No client-side server-only imports detected"

# Verify no sensitive credentials are exported from server-only modules
find packages/lib/server-only -type f -name "*.ts" -exec grep -l "export.*API_KEY\|SECRET\|PASSWORD" {} \; | wc -l

# Check client bundle size for anomalies
npm run build && du -sh dist/client/**/*.js | awk '{if($1 ~ /M/ && $1+0 > 5) print "Warning: Large client bundle detected: " $0}'
```

**Accept when:**
- All server-only modules reside under designated server-only directory structures and are not imported directly in client-side code
- Build process successfully prevents bundling of server-only code in client JavaScript outputs
- CI pipeline includes automated checks that fail on detection of server-only imports in client contexts
- All external API integrations, PDF operations, and audit log rendering are implemented as server-only modules
- ESLint rules enforce server-only import restrictions with no violations
- Bundle analysis confirms no server-only code appears in client chunks

<enforcement>
Claude Code MUST NOT skip or defer verification of server-only module boundaries. Violations of R-SERVERONLY-001 through R-SERVERONLY-008 block deployment and require immediate remediation. All security-sensitive changes require verification against these rules before approval.
</enforcement>
