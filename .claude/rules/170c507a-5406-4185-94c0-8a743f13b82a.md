<rule_activation id="170c507a-5406-4185-94c0-8a743f13b82a" title="Enforce Input Validation for Data Access Operations: Input Validation Logic" applies_to="packages/lib/server-only/**/*.ts">
These rules are ALWAYS ACTIVE for all data access operations involving user-supplied input parameters in server-only modules, including recipient updates, field modifications, and document operations.
</rule_activation>

### Rules

- **R-VAL-001** SHOULD: Input validation logic SHOULD be centralized in reusable validation utilities rather than duplicated across functions.

### Scope

**In scope:**
- All server-only data access functions in packages/lib/server-only
- Recipient management operations (create, update, delete)
- Field setting and modification operations
- Document and envelope mutation operations
- Any function accepting user-supplied parameters that interact with the database

**Out of scope:**
- Read-only query operations that do not modify data
- Internal system operations with hardcoded parameters
- Background jobs with validated input from trusted sources
- Migration scripts and administrative tools

**Exceptions:**
- EXC-001: Internal administrative functions with pre-validated input from trusted system components

### Verify

```bash
# Count validation-related patterns in server-only modules
grep -r 'validate\|sanitize\|check' packages/lib/server-only --include='*.ts' | wc -l

# Check for validation in recipient and field operations
grep -r 'function.*update.*recipient\|function.*set.*field' packages/lib/server-only --include='*.ts' -A 5 | grep -c 'validate\|throw.*Error'

# Run input validation tests
npm run test -- --grep 'input validation' --reporter json | jq '.stats.passes'
```

**Accept when:**
- All data access functions in server-only modules contain input validation logic before database operations
- Validation tests exist for each data access function covering valid, invalid, and edge case inputs
- Code review checklist includes verification of input validation for all new data access functions
- Centralized validation utility library exists (e.g., @lib/validation) with reusable validators
- Validation schemas are implemented using type-safe libraries (Zod, Yup, or equivalent)

<enforcement>
Claude Code MUST verify input validation is present in all data access functions before approving changes to packages/lib/server-only. Verification is mandatory and cannot be deferred. CI pipeline MUST fail if new data access functions lack validation tests. Code review MUST block merge if validation is missing or insufficient.
</enforcement>
