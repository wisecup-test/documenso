<rule_activation id="53eaee51-d8d2-41e8-9791-71502430ba3c" title="Standardize Environment Variable Configuration Management Pattern: Configuration Access Occur" applies_to="**/*">
These rules are ALWAYS ACTIVE for all runtime configuration and environment management implementations across server-side application code.
</rule_activation>

### Rules

- **R-CONFIG-001** MUST: Configuration access MUST occur through a dedicated configuration module or service that provides type-safe access to configuration values.

### Verify

```bash
# Check for direct process.env access outside configuration modules
grep -r 'process\.env\.' --include='*.ts' --include='*.js' packages/ | grep -v 'config\|environment' | wc -l

# Verify centralized configuration modules exist
find packages/ -name 'config.ts' -o -name 'environment.ts' -o -name 'env.ts' | head -5

# Check for configuration imports in API and library code
grep -r 'import.*config.*from' --include='*.ts' packages/api packages/lib | wc -l
```

**Accept when:**
- Direct process.env access is minimal or zero outside of dedicated configuration modules
- At least one centralized configuration module exists that exports typed configuration objects
- Configuration imports are present in API and library code indicating usage of centralized configuration

<enforcement>
Claude Code MUST verify configuration patterns through code review checklist, ESLint rules configured to warn on direct process.env access, and CI pipeline verification commands. Violations must be flagged in code review and resolved before merge to main branches. Verification is mandatory and cannot be skipped or deferred.
</enforcement>
