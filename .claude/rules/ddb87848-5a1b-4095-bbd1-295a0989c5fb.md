<rule_activation id="ddb87848-5a1b-4095-bbd1-295a0989c5fb" title="Adopt Server-Only Data Access Layer with Function-Based Repository Pattern: Data Access Operations" applies_to="**/*">
These rules are ALWAYS ACTIVE for all server-side data access operations. All database queries and data mutations MUST be implemented through the server-only data access layer.
</rule_activation>

### Rules

- **R-SODA-001** SHOULD: Data access operations SHOULD be organized by domain entity (e.g., field/, folder/, public-api/) within the server-only directory
- **R-SODA-002** MUST: All database queries and mutations for application data MUST be located in the server-only directory structure with no direct database queries in client-side code
- **R-SODA-003** MUST: Each data access function MUST follow the naming convention (get-*, find-*, update-*, delete-*, remove-*) and be isolated in its own file
- **R-SODA-004** MUST: Build process MUST successfully prevent client-side imports of server-only modules and fail with clear error messages when violations are detected
- **R-SODA-005** MUST: CRUD operations for domain entities (fields, folders, API tokens, etc.) MUST be implemented as server-side operations
- **R-SODA-006** MUST: Server-side API route handlers and server components MUST use the server-only data access layer
- **R-SODA-007** MUST: Background jobs and scheduled tasks requiring data access MUST use the server-only data access layer
- **R-SODA-008** MUST: Authentication and authorization checks requiring database queries MUST be implemented in the server-only layer

### Verify

```bash
# Check for client-side imports of server-only modules
grep -r 'from.*server-only' --include='*.tsx' --include='*.ts' apps/*/app/**/page.tsx apps/*/app/**/layout.tsx | grep -v '.server.' || echo 'No client-side imports of server-only modules detected'

# Count data access functions following naming convention
find packages/lib/server-only -type f -name '*.ts' | xargs -I {} basename {} | grep -E '^(get|find|update|delete|remove)-.*\.ts$' | wc -l

# Verify server-only directory exists and contains exported functions
test -d packages/lib/server-only && find packages/lib/server-only -type f -name '*.ts' -exec grep -l 'export.*function' {} \; | wc -l
```

**Accept when:**
- All database access operations are located in the server-only directory structure with no direct database queries in client-side code
- Each data access function follows the naming convention (get-*, find-*, update-*, delete-*, remove-*) and is isolated in its own file
- Build process successfully prevents client-side imports of server-only modules and fails with clear error messages when violations are detected
- Code review checklist includes verification that new data access operations follow the established pattern
- Automated ESLint rules are enforcing server-only module import restrictions in client code
- CI pipeline verification commands confirm pattern compliance

<enforcement>
Claude Code MUST NOT skip or defer verification of server-only data access layer compliance. All database operations MUST be verified to exist in the server-only directory. Build-time enforcement through bundler configuration is mandatory.
</enforcement>
