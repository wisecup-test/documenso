<rule_activation id="e954f1f9-fac4-4636-913f-e370c8fdb1f1" title="Standardize Structured Logging with Context Propagation for API and Server Operations: Log Entries Not" applies_to="**/*">
These rules are ALWAYS ACTIVE for all logging operations in API endpoints, server-only functions, and service clients.
</rule_activation>

### Rules

- **R-LOG-001** MUST NOT: Log entries MUST NOT contain sensitive information such as passwords, tokens, API keys, or personally identifiable information (PII) unless explicitly redacted.
- **R-LOG-002** MUST: All HTTP API endpoints exposed through Hono or similar frameworks MUST use structured logging with request context at entry and exit points.
- **R-LOG-003** MUST: Server-only business logic functions including organization management, user operations, and license validation MUST log operation start/completion with entity identifiers.
- **R-LOG-004** MUST: Service clients for external API integrations and inter-service communication MUST use structured logging.
- **R-LOG-005** MUST: Background jobs and scheduled tasks that perform business operations MUST use structured logging.
- **R-LOG-006** MUST NOT: Production code paths MUST NOT contain console.log or console.error statements; use structured logger instead.
- **R-LOG-007** MUST: Correlation IDs MUST be generated at API gateway/entry points and propagated through all downstream calls via request context or headers.
- **R-LOG-008** SHOULD: Use logging level conventions: ERROR for failures requiring intervention, WARN for degraded states, INFO for significant business events, DEBUG for detailed troubleshooting.
- **R-LOG-009** MAY: Performance-critical hot paths where logging overhead is measured to exceed 5% of operation time may request exception EXC-001.
- **R-LOG-010** MAY: Legacy code modules scheduled for deprecation within 3 months may request exception EXC-002.

### Verify

```bash
# Check for console.log/console.error usage in production code
grep -r 'console\.log\|console\.error' packages/ --include='*.ts' --include='*.js' | wc -l
# Expected: 0 or minimal - use structured logger instead

# Verify structured logger adoption in API and server-only code
grep -r 'logger\.(info|error|warn|debug)' packages/api packages/lib/server-only --include='*.ts' | wc -l
# Expected: widespread adoption across files

# Check for sensitive data patterns near logging statements
grep -r 'password\|token\|apiKey' packages/ --include='*.ts' -A 2 -B 2 | grep -i 'log' | wc -l
# Expected: 0 - no sensitive data in logs
```

**Accept when:**
- All API endpoints in packages/api contain structured logging calls with request context at entry and exit points
- Server-only functions in packages/lib/server-only log operation start/completion with entity identifiers for business operations
- No instances of console.log/console.error in production code paths (verified by linting rules)
- Automated security scanning detects no sensitive data patterns in log statements
- Log aggregation dashboard shows consistent structured log format across all services with parseable JSON
- Correlation IDs are present in all log entries and propagate through service boundaries

Claude Code MUST verify all rules in this file. Verification is mandatory before accepting logging implementations. Violations of R-LOG-001 and R-LOG-006 are treated as blocking issues.
</enforcement>
