<rule_activation id="f7e5dd07-c702-4583-a566-50cc69c7516a" title="Adopt Prisma ORM as Primary Datastore Access Layer: Complex Queries Use" applies_to="**/*">
These rules are ALWAYS ACTIVE for all database access patterns in tRPC server routers and data access layers. All new database queries MUST follow the patterns established herein.
</rule_activation>

### Rules

- **R-PRISMA-001** SHOULD: Complex queries SHOULD use Prisma's relation loading and filtering capabilities to minimize N+1 query problems.

### Verify

```bash
# Verify Prisma Client imports in tRPC server routers
grep -r 'from @prisma/client' packages/trpc/server --include='*.ts' | wc -l

# Verify Prisma query builder method usage
grep -r 'prisma\.' packages/trpc/server --include='*.ts' | grep -v 'node_modules' | wc -l

# Verify no direct database driver imports outside Prisma configuration
find packages/trpc/server -name '*.ts' -exec grep -l 'pg\|mysql\|sqlite' {} \; | grep -v prisma | wc -l
```

**Accept when:**
- All tRPC router files import PrismaClient and use it for database operations (verify command 1 shows imports in router files)
- Database queries use Prisma's query builder methods (findMany, findUnique, create, update, delete) rather than raw SQL (verify command 2 shows prisma method calls)
- No direct database driver imports (pg, mysql2, sqlite3) exist outside of Prisma configuration files (verify command 3 returns 0 or only shows Prisma-related files)

<enforcement>
Claude Code MUST verify all three conditions before accepting database access implementations. Violations must be flagged and refactored to use Prisma patterns or documented as approved exceptions.
</enforcement>
