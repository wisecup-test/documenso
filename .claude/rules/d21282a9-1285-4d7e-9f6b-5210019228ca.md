<rule_activation id="d21282a9-1285-4d7e-9f6b-5210019228ca" title="Adopt Structured Logging with Observability Context in Data Access Operations" applies_to="packages/lib/server-only/**/*.ts">
These rules are ALWAYS ACTIVE for all server-side data access operations including database queries, data mutations, and access pattern implementations.
</rule_activation>

### Rules

- **R-LOG-001** MUST NOT: Logging implementations MUST NOT log sensitive data (passwords, tokens, PII) in plain text without appropriate redaction.
- **R-LOG-002** MUST: All server-only data access modules (database queries, ORM operations, repository patterns) require structured logging with observability context.
- **R-LOG-003** MUST: CRUD operations for core business entities (folders, organisations, users, security settings) must include structured logging statements.
- **R-LOG-004** MUST: Data mutation operations (create, update, delete) that modify persistent state must be instrumented with logging.
- **R-LOG-005** MUST: Data retrieval operations that access sensitive or business-critical information must include contextual logging.
- **R-LOG-006** MUST: Internal API endpoints that perform data access on behalf of authenticated users must implement structured logging.
- **R-LOG-007** SHOULD: Use correlation ID propagation through middleware to ensure all data access operations within a request share the same correlation identifier.
- **R-LOG-008** SHOULD: Implement asynchronous logging libraries and log sampling for high-frequency operations to minimize performance overhead.
- **R-LOG-009** MAY: Performance-critical hot paths may be exempted where logging overhead is measured and documented to cause >5% performance degradation (EXC-001).
- **R-LOG-010** MAY: Legacy code modules scheduled for deprecation within 3 months may be exempted (EXC-002).

### Verify

```bash
# Count data access operations with logging instrumentation
grep -r "logger\|console\.log\|log\." packages/lib/server-only --include="*.ts" | grep -E "(query|find|create|update|delete|fetch)" | wc -l

# Count files importing logging utilities
find packages/lib/server-only -name "*.ts" -exec grep -l "import.*logger\|import.*log" {} \; | wc -l

# Run logging coverage test
npm run test:logging-coverage 2>&1 | grep "Data access operations with logging" | grep -oE "[0-9]+%"
```

**Accept when:**
- At least 90% of data access operations in server-only modules contain structured logging statements with contextual information
- All new data access code passes automated linting checks for required logging instrumentation
- Code review checklist confirms presence of appropriate logging for data access operations before merge approval
- Log aggregation dashboard shows consistent log volume and structure from all data access modules
- No sensitive data (passwords, tokens, PII) appears in plain text in log output
- Correlation IDs are present in logs for request tracing across data access boundaries
