<rule_activation id="0da95739-f21d-48d7-818c-a01066ef2fb0" title="Implement Cache Layer for Public API Route Handlers: Cache Invalidation Occur" applies_to="**/*">
These rules are ALWAYS ACTIVE for all authenticated Remix route handlers under _authenticated+ directory, organization-level routes (o.$orgUrl.*), team-level routes (t.$teamUrl+/*), and admin routes (admin+/*) handling settings, members, teams, groups, templates, documents, and support.
</rule_activation>

### Rules

- **R-CACHE-001** MUST: Cache invalidation MUST occur on any mutation operation that affects the cached data.

### Verify

```bash
# Check for cache implementation in authenticated route handlers
grep -r "cache" apps/remix/app/routes/_authenticated+ --include="*.tsx" --include="*.ts" | grep -E "(useCache|cacheGet|cacheSet)" | wc -l

# Check for cache invalidation logic in mutation operations
grep -r "invalidate" apps/remix/app/routes/_authenticated+ --include="*.tsx" --include="*.ts" | grep -E "(cache|Cache)" | wc -l

# Verify shared cache utility module exists
find apps/remix/app/utils -name "*cache*.ts" -o -name "*cache*.server.ts" | wc -l
```

**Accept when:**
- At least 80% of authenticated route handlers in the identified files implement cache layer for data retrieval operations
- A shared cache utility module exists and is imported by multiple route handlers
- Cache invalidation logic is present in mutation operations for cached resources
- Cache keys include scope identifiers (orgUrl, teamUrl, userId) to ensure tenant isolation

<enforcement>
Claude Code MUST NOT skip or defer verification. Cache invalidation bugs could lead to users seeing stale or incorrect data in multi-tenant scenarios. All mutation operations affecting cached data MUST trigger appropriate cache invalidation.
</enforcement>
